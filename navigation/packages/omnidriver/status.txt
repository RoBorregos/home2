ODrive Dashboard — Change Summary (2026-06-20)
===============================================

Ported the full odrive_dashboard node + dashboard.html from
home-custom-base's odrive_comm package, which is the upstream/reference
implementation of this same dashboard. For complete design detail on
everything below (the on-MCU EKF, the firmware error-reporting pipeline,
the CAN parameter/RxSdo feature, the full UART protocol, per-task firmware
flow diagrams, etc.), see the home-custom-base repo:

    https://github.com/RoBorregos/home-custom-base/tree/main

Specifically:
  - firmware/STM32H7_OMNIBASE_CAN_BNO085/omnibase_documentation.md
      section 7  — firmware error reporting system
      section 9  — UART command protocol reference (incl. sub_type 31)
      section 10 — telemetry format + CAN parameter access (RxSdo)
      section 11 — on-MCU EKF design detail
      section 12 — pending home2 omnidriver migration checklist (this
                   change closes most of that checklist)
  - omnibase_ws/src/odrive_comm/README.md — package-level docs for the
    upstream dashboard this one mirrors
  - omnibase_ws/src/odrive_comm/status.txt — the upstream package's own
    dated changelog (2026-06-20 entry covers the same EKF/error/RxSdo work)
  - STATUS.txt (repo root) — overall project status

NOTE: The dashboard's Axis Lab tab (per-motor inspection/control/
calibration, #tab-axislab in dashboard.html) is UNTESTED upstream — not
yet validated against real hardware. Carried over as-is; treat as
unverified here too.

Files changed (this package):
  - omnidriver/odrive_dashboard.py — replaced with the upstream
    odrive_comm implementation: on-MCU EKF quaternion/covariance
    consumption, odom -> base_link TF broadcast (publish_tf param,
    replaces robot_localization for the omnibase config), firmware
    error log parsing (E=/ELOST= lines -> firmware_error socket event),
    arbitrary CAN parameter writes (RxSdo, sub_type 31,
    send_set_param_float helper), numeric TELEM_FIELDS index-based
    parsing (matches the firmware's compacted wire format), SM_state
    tracking (distinguishes SM_IDLE from SM_ESTOP). home2-specific
    customizations preserved on top: serial_port default
    '/dev/ttyOmniSTM32', use_stamped_cmd_vel default False (nav2 Humble
    publishes unstamped Twist), node_ids default [36,34,33,40],
    imu_frame_id default 'base_link', linear_scale_x/y wheel-radius
    calibration (applied to the EKF's body-frame twist), the web-socket
    disconnect safety (zeroes commanded velocity), and the IMU yaw
    outlier pre-filter (defence-in-depth on top of the firmware's own
    EKF-side rejection).
  - assets/dashboard.html — replaced with the upstream odrive_comm
    version: adds the Axis Lab tab, the "Firmware errors" log panel
    (separate from the activity log), the "Advanced param (RxSdo)" box,
    VEL_RAMP as the default input mode (was Passthrough), and widened
    gain-slider limits (Pos -20..20, Vel/Int -10..10).

Requires: tf2_ros (already in package.xml's exec_depend).

Build + run:
    colcon build --packages-select omnidriver
    source install/setup.bash
    ros2 run omnidriver odrive_dashboard

Staged rollout reminder (see omnibase_documentation.md section 12.4):
flash the EKF firmware first, confirm this updated node works correctly
against it, THEN remove robot_localization from omni_basics.launch.py —
don't do both at once.

ODrive Dashboard — Change Summary (2026-05-06)
===============================================

1. BLUETOOTH STATUS DISPLAY
----------------------------
Files changed:
  - odrive_comm/odrive_dashboard_2.py
  - assets/dashboard.html
  - assets/dashboard_offline.html

What was done:
  - The node now parses BT_active from each incoming STM32 telemetry line.
  - bt_active (int 0 or 1) is added to the telemetry dict emitted over Socket.IO.
  - Col 3 of the top row now shows a read-only "Bluetooth control: Active / Inactive"
    indicator at the bottom of the card.
    Green = Active (BT_active=1), Red = Inactive (BT_active=0).
  - Updated purely by incoming telemetry — no button or toggle in the browser.


2. OFFLINE DASHBOARD
---------------------
Files changed:
  - assets/dashboard_offline.html  (new file)
  - assets/js/chart.umd.min.js     (user must download — see SETUP below)
  - assets/js/socket.io.min.js     (user must download — see SETUP below)
  - odrive_comm/odrive_dashboard_2.py  (two new Flask routes)
  - setup.py  (offline HTML and assets/js/ files added to data_files)

What was done:
  - dashboard_offline.html is identical to dashboard.html (including BT indicator)
    but uses local JS files instead of CDN links:
      CDN (online):  chart.js@4.4.0          → /js/chart.umd.min.js
      CDN (online):  socket.io@4.7.2         → /js/socket.io.min.js
    Both are served by a new GET /js/<file> Flask route from assets/js/.

  WHY NOT /socket.io/socket.io.js?
    python-socketio 5.x (installed: 5.16.1) dropped built-in client JS serving.
    The /socket.io/socket.io.js endpoint returns HTTP 400.
    The socket.io client must be bundled locally instead.

  - The /js/ Flask route searches two locations (first match wins):
      1. Installed share dir — install/odrive_comm/share/odrive_comm/assets/js/
      2. Source tree      — src/odrive_comm/assets/js/  (works without rebuilding)

  - Offline dashboard accessible at: http://localhost:5000/offline
  - Online dashboard (/) is unchanged.


REQUIRED SETUP — download JS files locally (internet required once)
--------------------------------------------------------------------
Run both commands from a machine with internet access:

    mkdir -p ~/Github/home-custom-base/omnibase_ws/src/odrive_comm/assets/js

    wget -O ~/Github/home-custom-base/omnibase_ws/src/odrive_comm/assets/js/chart.umd.min.js \
        https://cdn.jsdelivr.net/npm/chart.js@4.4.0/dist/chart.umd.min.js

    wget -O ~/Github/home-custom-base/omnibase_ws/src/odrive_comm/assets/js/socket.io.min.js \
        https://cdn.jsdelivr.net/npm/socket.io@4.7.2/client-dist/socket.io.min.js

Status:
  chart.umd.min.js   — DONE (confirmed present)
  socket.io.min.js   — PENDING (run the wget command above, then restart the node)

After downloading socket.io.min.js and restarting the node,
http://localhost:5000/offline will be fully functional without internet.
